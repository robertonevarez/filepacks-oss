---
title: Diffing
description: How filepacks performs structural comparison between two .fpk artifacts.
---

The current public `filepacks compare` command performs structural comparison only.

It compares the two manifests by path, size, and hash. There is no semantic comparison mode in the current public OSS CLI.

## Comparison input

```bash
filepacks compare <baseline> <candidate>
```

Both inputs must resolve to existing `.fpk` files.

## What compare checks

`compare` reads the manifest from each artifact and classifies differences as:

- added files
- removed files
- changed files

A file is considered changed when either of these differ for the same path:

- `size`
- `hash`

## Output shape

The CLI prints a stable line-oriented summary:

```text
Compare
baseline=/absolute/path/to/baseline.fpk
candidate=/absolute/path/to/candidate.fpk
ok=false
added=0
removed=0
changed=1
changed_file=summary.txt
```

When artifacts are identical, `ok=true` and the added, removed, and changed counts are all `0`.

## Exit behavior

- exit `0`: no structural differences
- exit `20`: one or more files were added, removed, or changed
- exit `1`: usage or file/path error

## Core API result

`@filepacks/core compare()` returns a structural result object with:

- `mode: 'structural'`
- `ok`
- `summary.added`
- `summary.removed`
- `summary.changed`
- `added[]`
- `removed[]`
- `changed[]`

Each changed file includes both left and right size/hash values.

## Why compare uses the manifest

Manifest-based comparison keeps the review surface explicit:

- no payload extraction is required
- paths are already normalized
- file lists are already sorted
- comparison behavior is consistent across environments

That makes `compare` a useful baseline-versus-candidate primitive for both humans and automation.

## What compare does not do

The current public CLI does not document or support:

- semantic comparison decisions
- eval-specific result states
- local baseline resolution
- registry-backed references
- JSON compare output
