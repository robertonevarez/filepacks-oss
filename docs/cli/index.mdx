The current public CLI is intentionally narrow.

Supported commands:

```bash
filepacks pack <input> --output <file>
filepacks inspect <file>
filepacks verify <file>
filepacks compare <baseline> <candidate>
filepacks push <artifact> --endpoint <url> [--open]
filepacks push <baseline> <candidate> --endpoint <url> [--open]
```

Unsupported commands are not part of the public OSS surface, including `show`, `import`, `registry`, `list`, `tag`, `baseline`, `unpack`, and `config`.

## Choose the right command

| If you want to... | Run |
| --- | --- |
| package a directory as one artifact | `filepacks pack <input> --output <file>` |
| read the artifact summary | `filepacks inspect <file>` |
| confirm the artifact is intact | `filepacks verify <file>` |
| see whether packaged files changed | `filepacks compare <baseline> <candidate>` |
| send artifacts to a temporary review endpoint | `filepacks push <baseline> <candidate> --endpoint <url>` |

## Global help

```bash
filepacks --help
```

For a clean-shell trial, use:

```bash
npx filepacks --help
```

Command-specific help is also available:

```bash
filepacks pack --help
filepacks inspect --help
filepacks verify --help
filepacks compare --help
filepacks push --help
```

## Command behavior summary

| Command | What it does | Exit behavior |
| --- | --- | --- |
| `pack` | Create a deterministic `.fpk` artifact from a directory | `0` on success, `1` on usage or file errors |
| `inspect` | Read artifact metadata from a `.fpk` file | `0` on success, `1` on usage or file errors |
| `verify` | Check payload files against the manifest | `0` when valid, `1` when invalid or on usage/file errors |
| `compare` | Structurally compare two artifacts | `0` when identical, `20` when different, `1` on usage/file errors |
| `push` | Send artifacts to a preview endpoint for temporary review handoff | `0` when the review handoff succeeds, `1` on usage/path/network/API errors |

## Output style

The CLI uses line-oriented plain text output by default. That keeps the commands readable in a terminal and stable for simple shell workflows.

Examples:

- `pack` prints `input=`, `output=`, `name=`, `digest=`, `files=`, `bytes=`
- `inspect` prints `path=`, `name=`, `version=`, `digest=`, `files=`, `bytes=`
- `verify` prints `ok=true` or `ok=false`
- `compare` prints `ok=`, summary counts, and per-file change lines
- `push` prints `mode=temporary_review` or `mode=temporary_compare_review`, `stored=false`, the endpoint, and a review URL

Add `--json` when an agent, CI job, or script needs structured output:

```bash
filepacks pack ./run-output --output ./run.fpk --json
filepacks inspect ./run.fpk --json
filepacks verify ./run.fpk --json
filepacks compare ./baseline.fpk ./run.fpk --json
filepacks push ./baseline.fpk ./run.fpk --endpoint http://localhost:3000 --json
```

JSON mode preserves the same exit-code behavior as human output. For example, `compare --json` still exits `20` when artifacts differ. `push --json` exits `0` when the temporary review handoff succeeds, even when the comparison itself reports differences.

`push` is experimental and means temporary review handoff. It verifies local artifacts, posts them to a preview review endpoint for the request, and can open `/app#review=...` with a small encoded result summary. It does not store artifacts, create registry records, create workspaces, or provide cloud sync.

## Recommended flow

Most users start here:

1. `pack` the output directory.
2. `inspect` the resulting artifact.
3. `verify` it before trusting or sharing it.
4. `compare` it against a baseline when you care about repeated-run drift.

Use [CLI workflows](/cli/workflows) for end-to-end patterns.

## Core library mapping

The CLI is a conservative wrapper around `@filepacks/core`:

- CLI `pack` -> `pack()`
- CLI `inspect` -> `inspect()`
- CLI `verify` -> `verify()`
- CLI `compare` -> `compare()`
- CLI `push` -> local `inspect()`/`verify()` plus a preview endpoint request

Use CLI `--json` when you need structured results from a shell command. Use the core package when you already run inside Node.js.
